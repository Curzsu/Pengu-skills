# STAR 亮点卡片正反例对照

> 本文件收录简历亮点卡片的正反例对照。生成卡片前扫对照，避免常见坑。
> 发现新典型坑回写本文件。

## 评判标准

好卡片满足：
- **紧凑版**：1-3句，含S+T+A+R核心
- **量化结果**：必须有可量化指标（RT/QPS/可用性/成本）
- **强动词开头**：主导/设计/优化/排查/实现
- **技术决策可见**：不只写做了什么，要体现为什么这么做
- **场景可信**：业务背景+触发事件具体

烂卡片的特征：
- 漏 Result（最常见）
- 结果不量化（"提升了性能"）
- 只写做了什么不写为什么
- 场景笼统（"某系统"）
- 动词弱（"参与"/"协助"）
- 技术堆砌无故事线

## 对照1：MQ消息不丢失

### ❌ 烂卡片
> 参与订单系统开发，使用RocketMQ处理订单消息，保证了消息不丢失。

**问题**：
- "参与"动词太弱，看不出贡献
- 没有触发事件（为什么需要保证不丢失）
- 没有技术决策（怎么保证的）
- 没有结果量化
- 场景笼统

### ✅ 好卡片

**紧凑版**：
> 主导订单系统MQ可靠性治理，针对线上偶发消息丢失问题，从生产端同步发送+ACK确认、Broker持久化、消费端手动ACK三环节加固，消息丢失率从0.1‰降至0，月均挽回订单损失约200单。

**详细STAR版**：
- **S（情境）**：电商订单系统使用RocketMQ异步处理订单消息，线上偶发消息丢失导致订单状态不一致
- **T（任务）**：定位消息丢失环节并彻底治理，保证订单消息0丢失
- **A（行动）**：通过日志排查定位到丢失发生在生产端网络抖动（未开ACK）和消费端自动ACK失败两个环节；生产端改为同步发送+重试3次+本地消息表兜底，消费端改为业务处理完成后手动ACK，死信队列接入告警
- **R（结果）**：消息丢失率从0.1‰降至0，月均挽回订单损失约200单，相关方案沉淀为团队MQ使用规范

**差异分析**：
- 触发事件具体（偶发消息丢失导致状态不一致）
- 技术决策可见（三环节加固 + 本地消息表 + 手动ACK）
- 结果量化（0.1‰ → 0，月均200单）
- 强动词（主导）
- 有沉淀（团队规范）

## 对照2：接口RT优化

### ❌ 烂卡片
> 优化了订单查询接口性能，使用了缓存和索引，提升了性能。

**问题**：
- "优化了"动词模糊
- 没有起点和终点（RT从多少到多少）
- "缓存和索引"泛泛而谈，无技术决策
- "提升了性能"是空话

### ✅ 好卡片

**紧凑版**：
> 排查并优化订单查询接口RT突增问题，定位到慢SQL（索引失效）+ 缓存击穿双因，通过重建联合索引 + 热点key永不过期+异步刷新方案，接口RT从2s降至80ms，P99 < 200ms，缓存命中率从60%提升至98%。

**详细STAR版**：
- **S（情境）**：电商大促期间订单查询接口RT从50ms飙到2s，触发告警
- **T（任务）**：48小时内定位根因并恢复RT到200ms以内
- **A（行动）**：通过APM链路追踪 + 慢SQL日志定位到两个根因：①WHERE条件字段类型不匹配导致索引失效；②大促热点订单key失效瞬间大量请求穿透到DB。前者重建联合索引并修正ORM映射，后者采用热点key永不过期+异步刷新（后台线程定时刷新）方案
- **R（结果）**：接口RT从2s降至80ms，P99 < 200ms，缓存命中率从60%提升至98%，方案推广到其他热点查询接口

**差异分析**：
- 双因定位（不是单一原因，体现排查深度）
- 具体技术方案（联合索引 + 永不过期 + 异步刷新）
- 多维量化（RT + P99 + 命中率）
- 有推广（方案价值外溢）

## 对照3：缓存击穿

### ❌ 烂卡片
> 在电商项目中解决了缓存击穿问题，使用了互斥锁。

**问题**：
- 没说为什么会有缓存击穿
- "使用了互斥锁"只说手段没说为什么选这个
- 没有结果
- 单句信息密度低

### ✅ 好卡片

**紧凑版**：
> 设计电商商品详情页缓存方案，针对大促爆款商品缓存击穿问题，对比互斥锁 vs 逻辑过期方案后选用逻辑过期（后台异步刷新），避免锁等待导致的线程阻塞，大促期间DB QPS峰值从8000降至500，无慢查询告警。

**详细STAR版**：
- **S（情境）**：电商商品详情页缓存击穿，大促爆款商品key过期瞬间大量请求打到MySQL
- **T（任务）**：设计缓存方案，防止击穿同时不影响大促流量
- **A（行动）**：对比互斥锁（实现简单但锁等待阻塞线程）和逻辑过期（无锁但短暂数据不一致），结合大促高并发场景选逻辑过期方案：value中存逻辑过期时间，请求发现过期后返回旧值并异步触发刷新
- **R（结果）**：大促期间DB QPS峰值从8000降至500，无慢查询告警，方案成为团队缓存设计默认选项

**差异分析**：
- 有方案对比（体现技术决策能力）
- 选型有理由（高并发场景怕锁等待）
- 结果多维（DB QPS + 无告警）
- 有推广

## 对照4：分布式锁

### ❌ 烂卡片
> 用Redis实现分布式锁，解决了秒杀超卖问题。

**问题**：
- "用Redis实现"无技术细节
- 没说为什么需要分布式锁
- 没说锁的边界（什么场景失效）
- 没有结果

### ✅ 好卡片

**紧凑版**：
> 主导秒杀系统防超卖方案，基于Redis + Lua脚本实现原子性库存扣减分布式锁，针对极端情况下锁续期失败问题引入Redlock+看门狗机制，压测10万QPS下无超卖，库存最终一致性与DB对账零差异。

**详细STAR版**：
- **S（情境）**：秒杀活动库存超卖10单，定位到并发扣减时多个线程同时读到旧库存
- **T（任务）**：设计高并发下防超卖方案，支持10万QPS
- **A（行动）**：用Redis + Lua脚本实现库存扣减的原子性（DECR + 判断在一个Lua中执行），避免SETNX锁的获取-判断-扣减非原子问题；针对锁续期失败场景引入Redisson看门狗自动续期；对账兜底用定时任务对比Redis库存与DB库存
- **R（结果）**：压测10万QPS下无超卖，库存最终一致性与DB对账零差异，方案推广到所有秒杀品类

**差异分析**：
- 有触发事件（超卖10单）
- 技术细节到位（Lua原子性 + 看门狗 + 对账兜底）
- 边界考虑（锁续期失败）
- 结果多维（压测QPS + 对账零差异）

## 对照5：慢SQL优化

### ❌ 烂卡片
> 优化了订单查询的慢SQL，加了索引。

### ✅ 好卡片

**紧凑版**：
> 排查订单查询慢SQL告警，通过EXPLAIN分析定位到深分页 + 多字段OR导致索引失效，改用游标分页（WHERE id > last_id LIMIT N）+ 拆分OR查询UNION ALL，慢SQL从1.2s降至30ms，慢查询日志条目归零。

**详细STAR版**：
- **S（情境）**：订单列表分页查询在深翻页时（page > 100）RT飙到1.2s，慢查询日志频繁告警
- **T（任务）**：优化深分页查询，RT降至100ms以内
- **A（行动）**：EXPLAIN发现深分页时OFFSET扫过的行数过多；同时多字段OR导致优化器放弃索引。前者改用游标分页（WHERE id > last_id LIMIT N）避免OFFSET，后者拆OR为多个子查询UNION ALL各自走索引
- **R（结果）**：RT从1.2s降至30ms，慢查询日志条目归零，方案沉淀到团队SQL规范

## 常见坑总结

| 坑 | 表现 | 修复 |
|----|------|------|
| 漏 Result | 卡片停在"做了什么" | 必须有量化结果（RT/QPS/可用性/成本） |
| 结果不量化 | "提升了性能" | 给具体数字（从X到Y） |
| 动词弱 | "参与/协助" | 用"主导/设计/优化/排查/实现" |
| 场景笼统 | "某系统" | 给具体业务模块 + 触发事件 |
| 无技术决策 | 只写做了什么 | 写为什么这么选（方案对比） |
| 技术堆砌 | 罗列技术名词 | 用故事线串起来（触发->排查->决策->结果） |
| 单因排查 | "定位到是XX问题" | 多因定位体现深度（双因/三因） |
| 无边界考虑 | 只讲成功路径 | 提及异常场景/兜底方案 |
| 结果无推广 | 卡片自洽即可 | 提方案沉淀/推广（团队规范/其他接口） |

## 量化指标参考库

写 Result 时按指标类型选：

| 指标类型 | 示例表述 |
|---------|---------|
| 响应时间 | RT从Xms降至Yms / P99 < Zms |
| 吞吐量 | QPS从X提升到Y / 单机承载QPS X |
| 可用性 | 可用性从99.9%提升至99.99% |
| 成本 | 服务器成本降低X% / 内存占用减少Y% |
| 数据量 | 处理X万条数据 / 单表Y亿行 |
| 错误率 | 错误率从X‰降至0 / 月均减少Y单损失 |
| 命中率 | 缓存命中率从X%提升至Y% |
| 业务影响 | 挽回订单损失X单 / 影响用户Y万 |
