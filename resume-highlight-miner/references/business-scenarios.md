## 电商后端

### 通用技术栈
Spring Boot + Spring Cloud + MySQL + Redis + RocketMQ/Kafka + Elasticsearch + Sentinel

### 业务模块与典型量级

| 模块 | 典型QPS | 典型问题 | 可包装能力 |
|------|---------|---------|-----------|
| 商品详情 | 1000-5000 | 缓存击穿/热点key | 高并发应对、缓存架构 |
| 订单创建 | 500-2000 | 超卖/幂等/分布式事务 | 并发控制、一致性 |
| 订单查询 | 1000-3000 | 分库分表/慢SQL | 容量规划、SQL调优 |
| 库存扣减 | 500-2000 | 分布式锁/原子性 | 并发控制、分布式 |
| 秒杀活动 | 10000-100000 | 限流/缓存/异步 | 高并发架构、限流设计 |
| 搜索 | 500-2000 | ES性能/相关性 | 系统设计、调优 |
| 购物车 | 500-1500 | 一致性/批量操作 | 数据一致性 |

### 可信触发事件库

- 大促期间订单接口RT从50ms飙到2s -> 性能调优/线上排查
- 秒杀活动库存超卖10单 -> 并发控制/分布式锁
- 双11缓存集群雪崩 -> 容错设计/高可用
- 商品上下架消息丢失导致展示不一致 -> MQ可靠性
- 慢SQL告警，订单查询接口超时 -> SQL调优
- 黑产刷接口查询不存在商品id -> 缓存穿透防御

### 项目骨架示例

**项目名**：XX电商平台后端
**角色**：订单服务/商品服务/秒杀服务 开发
**时间**：3-6个月（课设/实习/开源）
**团队**：3-5人
**技术栈**：Spring Boot + MySQL + Redis + RocketMQ

## 支付系统

### 通用技术栈
Spring Boot + MySQL + Redis + Kafka + 分布式事务框架（Seata/TCC）+ 监控（Prometheus）

### 业务模块与典型量级

| 模块 | 典型QPS | 典型问题 | 可包装能力 |
|------|---------|---------|-----------|
| 支付下单 | 200-1000 | 幂等/分布式事务 | 一致性、幂等设计 |
| 支付回调 | 200-800 | 重复回调/状态机 | 幂等、状态机设计 |
| 对账系统 | 定时任务 | 数据不一致 | 数据一致性、排查能力 |
| 退款 | 50-200 | 跨服务事务 | 分布式事务 |
| 风控 | 500-2000 | 规则引擎性能 | 系统设计、性能优化 |

### 可信触发事件库

- 支付回调重复到达，扣款两次 -> 幂等设计
- 跨服务扣款+扣库存不一致 -> 分布式事务
- 对账发现订单状态和支付状态不一致 -> 数据一致性排查
- 退款服务跨账户事务失败 -> TCC/Saga
- 风控规则引擎RT过高影响下单 -> 性能优化

### 项目骨架示例

**项目名**：XX支付系统
**角色**：支付服务/对账服务 开发
**技术栈**：Spring Boot + MySQL + Kafka + Seata

## 社交/IM

### 通用技术栈
Netty + WebSocket + Redis + Kafka/RocketMQ + MySQL + MongoDB

### 业务模块与典型量级

| 模块 | 典型QPS | 典型问题 | 可包装能力 |
|------|---------|---------|-----------|
| 消息发送 | 500-3000 | 消息顺序/不丢 | MQ可靠性、顺序性 |
| 消息推送 | 1000-5000 | 长连接/推送性能 | 长连接、性能优化 |
| 未读数 | 1000-3000 | 一致性/实时性 | 数据一致性 |
| 群聊 | 200-1000 | 消息扩散/存储 | 写扩散/读扩散 |
| 历史消息 | 500-2000 | 分库分表/查询 | 容量规划、分片 |

### 可信触发事件库

- 群聊消息乱序导致显示错误 -> MQ顺序性
- 消息推送偶发丢失 -> MQ可靠性 + 长连接
- 未读数不准，用户投诉 -> 数据一致性
- 历史消息查询慢，分库分表 -> 容量规划
- 大群消息写扩散存储压力大 -> 读写扩散权衡

### 通用技术栈
Spring Boot + MySQL + Redis + Vue/React + 权限框架

### 业务模块与典型问题

| 模块 | 典型问题 | 可包装能力 |
|------|---------|-----------|
| 多租户 | 数据隔离/资源隔离 | 架构设计、隔离方案 |
| 权限 | RBAC/数据权限 | 安全设计 |
| 工作流 | 状态机/审批流 | 状态机设计 |
| 报表 | 大数据量查询/导出 | 性能优化、分页 |
| 通知 | 多渠道/模板 | 模板设计、MQ |

### 可信触发事件库

- 多租户数据越权访问 -> 数据隔离方案
- 报表导出OOM -> 流式处理 + 异步
- 权限变更不生效 -> 缓存一致性

## 校园/实验室场景（校招专用）

> 校招/实习用户没有大厂线上经验时，用这些可信场景。

### 可信项目类型

| 类型 | 可信度 | 适合包装的能力 |
|------|--------|--------------|
| 课程设计（选课系统/图书管理/博客） | 高 | 基础架构、CRUD优化 |
| 实验室项目（导师课题/横向） | 高 | 算法应用、数据处理 |
| 个人开源项目（GitHub star < 100） | 中 | 全栈能力、自驱力 |
| 竞赛项目（ACM/数学建模/创新创业） | 高 | 算法、团队协作 |
| 小公司实习（创业公司） | 中 | 完整链路、独立开发 |

### 校园场景包装技巧

- **避免**：声称参与大厂核心业务、虚构线上QPS过万
- **推荐**：用"课程设计优化到QPS 500"、"实验室项目压测发现瓶颈"等可信表述
- **数据量**：课程设计数据量1万-10万合理；实验室项目10万-100万合理
- **触发事件**：用"压测发现"、"上线后发现"代替"线上告警"

### 校园项目骨架示例

**项目名**：XX课程设计 - 选课系统
**角色**：后端开发（独立/2人）
**技术栈**：Spring Boot + MySQL + Redis
**数据量**：学生1万、选课记录10万
**压测**：JMeter 压测选课高峰QPS 500

## 使用说明

1. 收到本质能力后，先选最贴合的业务方向（默认电商）
2. 在该方向找到对应的模块和可信触发事件
3. 套用项目骨架填充细节（项目名/角色/技术栈/量级）
4. 校招用户优先用校园场景骨架，避免虚构大厂经验
5. 构造完成后进入步骤3：STAR 包装（见 `star-card-examples.md`）
