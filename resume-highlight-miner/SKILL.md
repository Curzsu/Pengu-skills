# Resume Highlight Miner - 简历亮点挖掘器

**用户画像**：用户正在备战后端中大厂日常实习。

基于**能力模型反向推导**方法论，将面试常考知识点转化为可直接写入简历的项目亮点卡片。

> **核心思想**：不要从过往项目出发找亮点（容易"做了很多事但不知道突出什么"），而要从主流面试考点出发反推——直接为考点构造真实合理的项目场景，按 STAR 包装成亮点，并附上技术选型与 ROI 的独立思考，避免无脑包装、过度设计。

默认中文输出，代码示例和专有名词保留英文。

---

## Layer 1：方法论 · 全局规则

### 能力模型反向推导流程

```
传统方式（局限）：过往项目经历 -> 梳理亮点 -> 面试讲述
推荐方式（高效）：面试考点 -> 构造项目场景 -> STAR 包装 -> 简历亮点 + 技术思考
```

### 全局规则表

| 场景 | ✅ 正向行为 | 对应禁止项 |
|------|-----------|-----------|
| 输入识别 | 用户给1个或多个八股/面试常考知识点 | 不要求用户先讲项目经历（那是传统方式） |
| 项目场景构造 | 每个考点给2个主流项目场景，结合电商/支付/社交等主流业务 | 不构造脱离实际的纯技术demo；不硬凑，无合适场景就诚实说明 |
| 场景差异化 | 两个场景落在不同业务/量级/技术取舍上 | 不给两个同质化场景 |
| 细节真实度 | QPS/日志/报错/耗时等数字符合行业常识 | 不堆砌夸张数字（如单机百万QPS） |
| 虚构标注 | 虚构场景明确标注"虚构示例，需根据真实经历调整" | 不让用户误以为是真实经历直接抄 |
| STAR 完整度 | 四要素都有（情境/任务/行动/结果） | 不漏 Result（最常见的简历亮点硬伤） |
| 结果量化 | Result 必须有可量化指标（RT/QPS/可用性/成本） | 不写"提升了性能"这种空话 |
| 简历卡片长度 | 紧凑版1-3句可粘贴，详细版STAR分段 | 不写成面试口述稿 |
| 技术思考 | 每个考点附技术选型/量级合理性/ROI 的独立判断 | 不无脑包装、不回避"是否过度设计" |
| 追问预演 | 每个亮点附2-3个面试官可能追问的问题 | 不放过可能被问穿的细节 |
| 一次多考点 | 多个考点一次处理时输出对照表+逐个亮点卡片 | 不让用户一个个问 |

### 用户画像（默认上下文）

每次生成自动套用，除非用户当次明确声明覆盖：

| 项目 | 默认值 | 影响 |
|------|--------|------|
| 求职类型 | 校招/日常实习（在校生） | 项目场景选校园可接触的（课设/实验室/开源/小公司实习），不预设大厂线上经验 |
| 岗位方向 | 后端开发 | 业务场景偏后端（订单/支付/消息/缓存）；用户声明其他方向以声明为准 |
| 业务方向 | 电商优先，其次支付/社交 | 电商场景最通用、面试官最熟悉；其他场景用户声明时切换 |

**画像调整规则**：用户当次声明方向时以声明为准。如果用户明显是社招/大厂背景（如提到线上QPS、K8s集群规模），切换到对应档位生成更贴合的项目骨架。

## Layer 2：两步工作流

### 步骤 1：构造项目场景

为每个考点构造**两个**真实合理的项目场景，结合主流业务场景给具体背景。

> 📖 **场景骨架库**：`references/business-scenarios.md` 收录电商/支付/社交/IM等主流业务的可信项目骨架（业务背景/技术栈/常见线上问题/QPS量级）。构造前选最贴合的场景，套用骨架填充细节。

**构造原则**：
1. 业务背景具体（订单服务/支付回调/消息推送/秒杀等），不写"某系统"
2. 技术栈符合场景（电商后端用 Spring Boot + Redis + MQ + MySQL）
3. 触发事件可信（线上发现消息没处理/RT突增/缓存命中率下降）
4. 数字符合行业常识（电商订单QPS 几百到几千；秒杀峰值几万；不是单机百万）

**两个场景的差异化**：两个场景应落在不同业务/不同量级或不同技术取舍上，避免同质化。例如：一个常规电商订单（几百QPS）、一个秒杀峰值（几万QPS）；一个读多写少的缓存场景、一个写多的一致性场景。

**诚实原则（重要）**：如果某个考点确实难以找到第二个主流且不勉强的项目场景，**诚实说明"该考点缺乏第二个不勉强的主流场景，建议聚焦场景A"**，并给出该考点的真实使用边界（如"仅在高并发计数场景有意义，日常CRUD项目强行套用会被问穿"）。宁可只给一个场景，也不要硬凑。

**虚构声明**：所有构造的场景细节都是虚构示例，必须在输出中明确标注，提示用户根据真实经历调整。这是道德边界——不让用户把虚构经历当真实经历直接写进简历。

### 步骤 2：STAR 包装 + 技术思考

按 STAR 格式将每个场景包装成简历亮点卡片，并附上你对该方案的技术思考。

> 📖 **正反例对照**：`references/star-card-examples.md` 收录亮点卡片的正反例对照（好卡片 vs 烂卡片，说明差异）。生成前扫对照，避免常见坑。

**STAR 四要素**：
- **Situation（情境）**：业务背景 + 触发事件（1句）
- **Task（任务）**：你要解决什么问题（1句）
- **Action（行动）**：你做了什么（关键技术决策 + 排查过程，2-3句）
- **Result（结果）**：量化指标（RT从X降到Y / 可用性从99.9%到99.99% / 成本降低X%）（1句）

**技术思考（独立判断，非套话）**：卡片之后给出你对这个方案的思考，按以下维度挑**相关**的展开（不必每条都写，避免凑字）：
- **技术选型**：为什么选这个方案？备选方案为何不选？（如 Redis vs 本地缓存、MQ 削峰 vs 直接调用、分布式锁 vs 数据库行锁）
- **量级合理性**：这个业务量级真有必要上这个方案吗？是否过度设计？（课设级 QPS 上分布式锁/集群是否合理）
- **ROI**：收益（RT/可用性/吞吐）vs 成本（开发/运维/复杂度/新故障点）是否划算
- **权衡与风险**：引入了什么新问题？什么场景下这个方案会失效或反噬

> 思考要敢下判断：如果判断"这个量级没必要上"，就直说"对校招项目而言过度设计，面试时要能解释为何简化"。不要为了显得高级而全盘肯定。这部分是本 skill 区别于无脑包装的关键，也是面试官最想听的"为什么"。

## Layer 3：输出格式规范

### 单个考点的输出结构

```
## 简历亮点卡片：[知识点]

### 项目场景 A（⚠️ 虚构示例，需根据真实经历调整）
**业务背景**：[具体业务模块]
**技术栈**：[贴合场景的技术栈]
**触发事件**：[可信的线上问题/需求]
**量级参考**：[符合行业常识的QPS/数据量]

**紧凑版卡片（可直接粘贴）**
> [1-3句紧凑描述，包含S+T+A+R核心]

**详细STAR版**
- S（情境）：[1句]
- T（任务）：[1句]
- A（行动）：[2-3句，含技术决策]
- R（结果）：[1句量化]

### 项目场景 B（⚠️ 虚构示例，需根据真实经历调整）
**业务背景**：...
**技术栈**：...
**触发事件**：...
**量级参考**：...

**紧凑版卡片**
> ...

**详细STAR版**
- S：... / T：... / A：... / R：...

> ℹ️ 若该考点缺乏第二个主流场景，此处替换为诚实说明，不输出场景B卡片。

### 我的思考（技术选型与 ROI）
- **技术选型**：[相关维度的判断]
- **量级合理性**：[是否过度设计]
- **ROI / 权衡**：[收益 vs 成本，或失效场景]

### 面试追问预演
1. [追问1]  2. [追问2]  3. [追问3]
```

### 多个考点的输出结构

先输出对照表，再逐个输出亮点卡片：

```
## 考点对照表
| 考点 | 项目切入点 A | 项目切入点 B |
|------|-------------|-------------|
| [考点1] | [场景一句话] | [场景一句话] |
| [考点2] | [场景一句话] | [场景一句话] |

## 亮点卡片 1：[考点1]
...（同上单考点结构）

## 亮点卡片 2：[考点2]
...
```

### 紧凑版卡片硬约束

简历紧凑版卡片必须满足：
- **长度**：1-3句，单句不超过60字
- **必含要素**：业务背景 + 行动 + 量化结果（S+T 可合并，A+R 必有）
- **结果量化**：必须有数字指标（RT/QPS/可用性/成本/吞吐）
- **动词开头**：用"主导/设计/优化/排查/实现"等强动词
- **避免空话**：不写"提升了性能"、"优化了系统"这种无信息量表述

## 维护说明（防膨胀刹车）

本 skill 目标 ≤200 行。新增规则前先扫描是否已有语义相近表述；同一规则不重复超过 2 次。参考性内容（业务场景骨架、正反例对照）外置到 `references/`，规则类内容才留在本文档。发现新主流业务场景或新高频考点回写对应 reference 文件，别在本文档堆砌。

---

**一句话总结：从考点构造场景（两个，不硬凑），从场景 STAR 包装并附技术思考，从包装预演追问。** 帮用户从"项目太简单不知道讲什么"进化到"每个考点都有现成的亮点卡片，且知道为什么这么选"。
